# task-recorder

`task-recorder` 是一个可移植的 AI Skill，用于把长期、重要或容易中断的 AI 工作变成“可管理、可记录、可续接”的任务。

它会让 AI 在执行前先解释理解和规划，经过用户确认后再执行；执行过程中维护持久化 Markdown 任务记录，保存目标、范围变化、阶段进度、关键决策、验证结果和交接信息，让未来的新会话可以直接继续，不需要重新解释全部上下文。

该 Skill 适用于支持 Markdown Skill 的 AI CLI 和助手，例如 Claude Code 和 OpenCode。

## 核心价值

AI 长任务常见的问题是：上下文越来越长、需求中途变化、进度散落在聊天里、AI 忘记之前的决策、新会话无法安全续接。`task-recorder` 的目标就是让任务记录成为稳定的交接点。

主要优点：

- 可靠续接：新会话可以从 `.tasks/*.md` 继续，而不是重新讲一遍。
- 更安全执行：重要工作前必须先输出理解和规划，并等待用户确认。
- 节省上下文：最新状态放在文件顶部，历史按阶段追加。
- 更清楚的变更记录：每个阶段记录自己的变更、验证、结果和交接说明。
- 范围演进不覆盖历史：扩展和修订会追加到阶段日志，不会抹掉早期目标和结果。
- 重要产物持久化：实质内容应写入文件或外部产物，而不是只输出在聊天里。
- 多语言友好：记录和交互使用用户语言，同时保留代码、命令、路径、日志和技术术语原文。
- 便携：真正需要安装的核心只有一个 `SKILL.md` 文件。

## 记录什么

用户确认计划后，任务记录默认写入 `.tasks/`。

任务记录采用轻量结构：

- `AI 续接说明`：告诉未来 AI 如何安全恢复任务。
- `元数据`：状态、时间、工作区、记录路径、分支/提交引用。
- `概览`：当前事实源，包含状态、总体目标、当前阶段、下一步、阻塞和关键产物。
- `目标与范围历史`：保留原始目标、当前目标、新增范围和被替代范围。
- `用户需求`：累计约束、偏好和明确非目标。
- `预期产物`：任务预期生成的文件或外部产物。
- `关键决策`：重要决策和理由。
- `阶段日志`：追加式历史。每个阶段记录请求、范围关系、计划、进度、变更、验证、结果和交接。
- `续接摘要`：给下一次会话的简短交接说明。
- `最终总结`：整个任务完成或失败时填写。

## 触发方式

### 强触发

强触发会立即进入 task-recorder 流程。

在消息开头使用明确任务前缀：

```text
task: implement a CLI parser
Task: refactor the auth flow
任务：实现一个 Unity C# 有限状态机
开始任务：整理项目发布流程
继续任务：.tasks/unity-fsm.md
读取任务：.tasks/release-plan.md
```

提供 `.tasks/*.md` 任务记录也会触发恢复流程。

### 弱触发

弱触发只询问是否启用，不会直接创建任务记录，也不会直接通过 Skill 执行。

适用场景：任务复杂、需要续接，或用户语义上想保留进度、创建/维护任务记录、把当前工作转成托管任务、方便以后继续。

示例：

```text
请保留当前进度，方便后续新会话继续。
```

预期 AI 行为：

```text
这看起来适合使用任务记录来管理。

1. 使用 task-recorder - 确认后创建托管记录。
2. 正常继续 - 不创建任务记录。

请回复 1 或 2。
```

### 不触发

普通问答、闲聊、一步小改动和简单解释应正常处理，除非用户明确要求记录任务。

## 常见使用流程

### 新建托管任务

1. 用户使用 `task:` 或其他任务前缀开始。
2. AI 只做必要检查。
3. AI 输出理解、规划、预期产物和任务记录路径。
4. 用户确认或提出修改。
5. 用户确认后，AI 才创建 `.tasks/task-name.md` 并开始执行。
6. AI 在关键节点更新任务记录。

### 无前缀的复杂任务

1. 用户提出复杂或可续接工作，但没有使用任务前缀。
2. AI 询问是否使用 `task-recorder`。
3. 用户选择启用后，进入正常确认流程。
4. 用户拒绝时，AI 不创建任务记录，按普通方式继续。

### 对话中途采纳

如果工作已经在没有 Skill 的情况下开始，用户后来希望保留进度，AI 应询问是否把当前工作纳入 `task-recorder`。

用户同意后，AI 总结当前上下文、标注不确定内容、再次请求确认，然后创建任务记录。之前未记录的内容应标记为“基于当前上下文的尽力重建”，不能假装从一开始就完整跟踪。

### 从已有记录继续

给 AI 一个 `.tasks/*.md` 路径或文件：

```text
继续任务：.tasks/unity-fsm.md
```

AI 应先总结状态、目标、已完成、下一步和阻塞，再继续。

## 安装

运行时只需要安装 `SKILL.md`。本仓库中的其他文件是给人阅读和 GitHub 展示用的文档，不是 Skill 运行所必需。

推荐方式：

1. clone 整个仓库，方便阅读文档和后续拉取更新。
2. 将 `SKILL.md` 复制或软链接到目标 skills 目录。

最小安装方式：

1. 只下载或复制 `SKILL.md`。
2. 放到目标 skills 目录。

README、许可证和变更记录不需要安装到 skills 目录。如果你把整个 `task-recorder` 文件夹复制到 skills 目录，大多数 skill 加载器通常也只会使用 `SKILL.md`，但只安装 `SKILL.md` 是最干净的方式。

### Claude Code

复制或软链接 `SKILL.md` 到：

```text
~/.claude/skills/task-recorder/SKILL.md
```

### OpenCode

复制或软链接 `SKILL.md` 到：

```text
~/.config/opencode/skills/task-recorder/SKILL.md
```

Windows 下 OpenCode 全局路径通常是：

```text
C:\Users\<you>\.config\opencode\skills\task-recorder\SKILL.md
```

安装或更新后需要重启 CLI。

## 测试方式

安装并重启 CLI 后，测试强触发：

```text
任务：实现一个小工具并记录过程
```

预期行为：

- AI 输出理解和规划。
- AI 等待用户确认。
- 用户确认后，AI 创建 `.tasks/*.md` 并开始执行。

测试弱触发：

```text
请保留当前进度，方便后续新会话继续。
```

预期行为：

- AI 询问是否启用 `task-recorder`。
- 用户选择启用前，AI 不创建记录。

## 设计原则

- 重要执行前先询问和确认。
- 弱触发只询问，用户确认前不创建任务记录。
- 最新状态放在记录顶部。
- 历史按阶段追加，不覆盖早期目标、计划和结果。
- 记录关键事实、决策、验证和交接信息。
- 避免无意义工具调用流水账。
- 重要输出优先保存为持久化产物。
- 保留代码、命令、路径、日志和错误信息的精确文本。
